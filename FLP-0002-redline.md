# FLP-0002: Redline Operation Vocabulary
**Version:** 1.0.0-draft  
**Status:** Draft  
**Depends on:** FLP-0000, FLP-0001

---

## 1. Abstract

This spec defines the vocabulary for expressing document changes (redlines)
between any two versions of a Folio-enabled document. The redline format
operates on the **Pandoc AST** — the format-agnostic semantic representation
defined in FLP-0000 and FLP-0004. It does not operate on raw format bytes.

This means redlines work identically whether the documents being compared
are DOCX, ODT, PDF, Markdown, or any other Pandoc-supported format.

---

## 2. Location Addressing

Every operation MUST include a `loc` attribute identifying where in the
document the change occurred. The addressing scheme uses human-readable
semantic paths derived from the document's structure, not format-specific
element references.

### 2.1 Location Format

```
{section-label}, {block-type} {block-index}
```

Examples:
```
Section 4.2, paragraph 1
Section 7, paragraph 3
Table 2, row 4, cell 2
Header
Footer
Preamble, paragraph 1
Paragraph 14
```

Section labels are derived from the document's heading structure as
represented in the Pandoc AST. If a document has no headings,
implementations use `paragraph {n}` with a global index.

### 2.2 Location Stability

Location addresses are computed at the time of redline creation and
are INFORMATIONAL — used for human display and conflict detection.
They reference the AST structure, not format-specific markup.
If a document is substantially restructured, existing location addresses
may no longer map precisely to the current document.

## 3. Operation Types

### 3.1 `modify` — Content changed in place

```json
{
  "type": "modify",
  "loc":  "Section 4.2, paragraph 1",
  "old":  "payment due net-30 from invoice date",
  "new":  "payment due net-60 from invoice date"
}
```

| Field  | Required | Description                              |
|--------|----------|------------------------------------------|
| `type` | MUST     | `"modify"`                               |
| `loc`  | MUST     | Location address                         |
| `old`  | MUST     | Previous text (verbatim excerpt from AST)|
| `new`  | MUST     | Replacement text                         |

### 3.2 `insert` — New content added

```json
{
  "type": "insert",
  "loc":  "Section 7, after paragraph 3",
  "new":  "Governing law shall be the State of Delaware..."
}
```

| Field  | Required | Description                                           |
|--------|----------|-------------------------------------------------------|
| `type` | MUST     | `"insert"`                                            |
| `loc`  | MUST     | Location including anchor (`after paragraph N`)       |
| `new`  | MUST     | The inserted content                                  |

### 3.3 `delete` — Content removed

```json
{
  "type": "delete",
  "loc":  "Section 12.4",
  "old":  "Liquidated damages: in the event of breach..."
}
```

| Field  | Required | Description          |
|--------|----------|----------------------|
| `type` | MUST     | `"delete"`           |
| `loc`  | MUST     | Location address     |
| `old`  | MUST     | The deleted content  |

### 3.4 `format` — Formatting changed, content unchanged

```json
{
  "type":        "format",
  "loc":         "Section 3, paragraph 2",
  "description": "Paragraph style changed from body to heading level 3"
}
```

Format operations are OPTIONAL. Implementations MAY omit them when
only semantic content comparison is needed. They SHOULD be included
when formatting is legally or structurally significant.

Note: because Folio fingerprints via Pandoc AST (which does not
capture all formatting), some formatting changes will not affect
the fingerprint. This is documented in FLP-0004.

### 3.5 `move` — Content relocated

```json
{
  "type":    "move",
  "from":    "Section 8, paragraph 4",
  "to":      "Section 3, after paragraph 1",
  "content": "The indemnification obligations herein shall survive..."
}
```

---

## 4. Redline Generation

Redlines are generated by:

1. Converting both documents to Pandoc AST JSON (FLP-0004 pipeline)
2. Applying a semantic diff algorithm (Myers diff or equivalent) to
   the block sequences of both ASTs
3. Expressing differences as a sequence of ops using this vocabulary

The diff operates on Pandoc AST **blocks** (Para, Header, Table,
BulletList, etc.) — not on raw text strings or format-specific markup.

---

## 5. Storage

Redline ops are stored within `<markup>` objects in the Folio record
(see FLP-0003, `collaboration.markups[].ops`).

They are also generated on-demand by conforming tools when a user
requests a comparison between any two versions or two documents.

---

## 6. Conflict Detection

Two markups CONFLICT if they contain operations with overlapping `loc`
values that produce incompatible results on the same base version.

```json
// Markup A: tax counsel
{ "type": "modify", "loc": "Section 8, paragraph 2",
  "old": "indemnification cap of $500,000",
  "new": "indemnification cap of $1,000,000" }

// Markup B: employment counsel  
{ "type": "modify", "loc": "Section 8, paragraph 2",
  "old": "indemnification cap of $500,000",
  "new": "indemnification cap shall not apply to employment claims" }
```

Same `loc`, different `new` values — this is a Dispute (FLP-0003).
Conforming tools MUST detect this before incorporation.

---

## 7. Human-Readable Summary

Conforming tools SHOULD generate a plain-language summary:

```
3 changes since version 1

MODIFIED
  Section 4.2 — Payment terms changed from net-30 to net-60

ADDED  
  Section 7 — New governing law clause (Delaware, binding arbitration)

REMOVED
  Section 12.4 — Liquidated damages clause deleted
```

AI-assisted implementations MAY generate more detailed narration
from the op log. See FLP-0017 (planned) for the AI narration interface.
